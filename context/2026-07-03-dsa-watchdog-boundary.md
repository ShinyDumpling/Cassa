# DSA / Watchdog / 预期与盯盘边界

## 主题概述

这一阶段的核心讨论，不是继续给旧项目叠功能，而是先重新厘清三个概念：

1. `DSA` 负责什么
2. `Watchdog` 负责什么
3. “预期”和“盯盘”之间怎么拆分

## 当前痛点

之前的组合方式大致是：

1. `DSA` 先生成大盘和个股报告
2. 再把这些报告和实时盯盘数据喂给 `Watchdog`
3. `Watchdog` 再次调用 LLM 生成策略或指导

问题在于：

1. 同一条决策链上调用了两次 LLM，流程长、慢、且边界模糊。
2. `Watchdog` 一边监控一边分析，职责不纯。
3. `DSA` 的输出有帮助，但股票一多时速度太慢，不适合实时环节。

## 已形成的边界认识

当前比较认可的拆分方向是：

### 1. 预期层

负责回答：

- 明天/下一阶段重点看什么方向
- 哪些板块是主线、轮动、避开方向
- 某只个股如果参与，应该以什么方式看待
- “如果发生什么，就怎么应对”的条件化预案

这个层更适合依赖 LLM，因为需要综合较多上下文、允许带有主观判断。

### 2. 盯盘层

负责回答：

- 现在发生了什么
- 是否触发预设提醒
- 哪些关键条件已经满足或失效

这个层第一版更倾向于不用 LLM，先用纯逻辑做快速监控和提醒。

## 关于“盘中预期”

讨论后认为，A 股风格变化快，只有盘后预期是不够的。除了盘前/盘后预期，还应该考虑：

1. 盘前预期
2. 盘中预期修正
3. 盘中盯盘提醒

但第一阶段优先顺序不是三块一起上，而是先把“预期”的生成逻辑理顺，再考虑 `Watchdog` 如何消费这些预期。

## 关于板块预期与个股预期

当前认可的关系是：

1. 板块预期由专门的大盘/板块模块产出
2. 个股预期不需要对每只股票重新从零推板块
3. 如果某只股票所属板块不在热点或重点观察板块内，可以直接标记为“缺少板块预期支持”
4. 个股预期应该结合：
   - 个股自身走势
   - 所属热点板块上下文
   - 个股在板块中的角色

## 关于个股角色的认知

讨论中逐渐形成一个思路：在个股预期里，股票不只是“强/弱”，还需要判断它在板块里的角色，例如：

1. 龙头
2. 中军
3. 弹性票
4. 跟风票

这个角色判断会直接影响后续预期表达和盯盘方式。

## 对旧方案的反思

还讨论过一条思路：先在 `DSA` 上做“个股预期”改造，复用它现有的数据和 LLM 流程。但做到后面发现：

1. 脚本越来越多
2. 中间 JSON / prompt / report / runner 分裂严重
3. 使用链路复杂
4. 很多东西虽然能跑，但整体体验越来越乱

这也是后来决定单独新开 `Cassa` 的直接原因之一。

## 当前结论

现阶段最重要的不是继续在旧项目上拼功能，而是先把以下边界写清楚：

1. 预期是什么
2. 盯盘是什么
3. 两者如何衔接
4. 哪部分必须由 LLM 判断，哪部分先交给规则逻辑
