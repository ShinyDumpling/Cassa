# Cassa 项目初始化与协作规则

## 主题概述

`Cassa` 是为了替代当前分散在 `DSA`、`Sentinel`、`alphasift-fork` 中、业务边界越来越混乱的一组脚本而新开的项目。第一阶段目标不是做一个“大而全”的系统，而是先用一个大脚本把核心流程重新捋顺。

## 为什么新开项目

之前的主要问题有三类：

1. 业务流程被拆成很多脚本，执行链长，中间产物多，容易过期，也不方便使用。
2. 控制台输出和结果文件过多，肉眼不容易快速抓重点。
3. 旧项目本身复杂，且混合了很多历史改造，导致业务流程越来越黑盒，后续继续叠功能风险很高。

因此决定：

1. 新开一个项目目录 `D:\股神养成plan\Cassa`
2. 第一版采用单脚本入口
3. 先把流程理顺，再考虑拆模块

## 命名与定位

- 项目代号最终定为 `Cassa`
- 主脚本文件名定为 `cassa.py`
- `README.md` 负责说明项目目标和范围
- `AGENTS.md` 负责约束 Agent 的协作与编码行为
- `context/` 负责保存项目上下文摘要

- `context/` 放会话整理后的上下文

## 协作原则

已经明确写入 `AGENTS.md` 的核心原则包括：

1. 修改代码前必须先讨论方案，并得到确认。
2. 未经允许不能提交代码。
3. 新会话第一次接触项目时，先看 `AGENTS.md` 和 `context/`。
4. 第一版优先简单、可读、可跑，不优先追求抽象和复用。
5. 输出要分层，默认不打印大段原始 JSON、Prompt、模型响应。

## 编码风格偏好

用户明确希望代码尽量满足以下要求：

1. 变量名、函数名一眼能看懂用途。
2. 函数都要有说明性注释或 docstring。
3. 重要变量、路径变量、状态变量需要注释。
4. 文件内尽量先集中定义常量/配置，再定义函数，再写主流程。
5. 第一版尽量少拆模块，先把主流程写清楚。

## context 书写规范（2026-07-06 更新）

每个 context 文件必须尽可能详细，目标是让新 Agent 在不问用户的情况下也能接上之前的设计和状态。内容至少应包含：

1. **主题概述**：这次讨论做了什么。
2. **功能实现**：最终选定的方案是什么、为什么这么选。
3. **代码设计**：新增/修改了哪些函数、数据结构、CLI 入口，它们的职责是什么。
4. **已完成内容**：本次实现了哪些功能点。
5. **TODO / 未做内容**：明确列出哪些东西还没做、为什么没做、下一步可以怎么做。
6. **数据源与接口坑点**：使用了哪些接口、字段名和单位有什么注意事项。
7. **测试验证**：用什么命令验证的、输出什么结果。
8. **与其他模块/系统的关系**：例如复用了哪些现有函数、与 DSA 等旧项目的对应关系。
9. **关键决策与反思**：做过哪些权衡、最后为什么这么决定。
10. **输出示例**：如果有 CLI 输出，一定要留一个典型输出示例。
11. **来源会话**：会话 ID 或日期。

如果某个旧结论已失效，必须在对应 context 文件中明确标注"已失效""已废弃"或"仅供参考"。

## 终端与编码问题

本轮还确认了一个环境问题：

1. 当前 Codex 会话里的旧 `powershell` 读取中文 UTF-8 文件时，`Get-Content` 容易出现乱码。
2. 用户本机已经安装 `PowerShell 7`。
3. 已经把 Windows Terminal 的默认配置改成 `PowerShell 7`，但这只对新开的终端生效，不影响当前 Codex 会话。
4. 当前会话里如果要正确读取中文文件，需要显式用 `pwsh -NoLogo -NoProfile -Command ...` 来读取。

## 当前结论

`Cassa` 项目现阶段不是要“马上复刻全部旧能力”，而是要先建立一个：

1. 边界清楚
2. 输入输出清楚
3. 中间产物可控
4. 新 Agent 容易接手

的最小新骨架。
