# 丁圣元趋势分析 Skill 新建完整方案

## 1. 文档目的

本文档定义如何基于 `book-to-skill` 已生成的《趋势投资》知识型 Skill，新建一个可接收A股代码、自动调用 Cassa 命令获取日K数据、再依据丁圣元《趋势投资》完成趋势分析的业务型 Agent Skill。

目标新Skill为 `C:\Users\super\Downloads\ding-trend-analyze`。原 `C:\Users\super\Downloads\ding-trend-investing` 保持为 book-to-skill 生成的学习型Skill，不作为本次业务Skill继续改写。业务入口是 `ding-trend-analyze/SKILL.md`；Skill在业务流程中使用以下现有命令获取日K数据：

```powershell
PS D:\股神养成plan\Cassa> python business.py thises --codes 002422
```

## 2. 项目背景

现有 `ding-trend-investing` 由 `book-to-skill` 从丁圣元《趋势投资：金融市场技术分析指南》EPUB生成，当前主要用途是：

- 回答书中概念。
- 按章节学习书中框架。
- 查询趋势、蜡烛图、价格形态等知识。

新的业务目标不是继续制作一个问答学习型知识库，而是从其知识产物派生出 `ding-trend-analyze`，让新Skill成为A股日K趋势分析的主要业务入口：

1. 用户提供一个或多个A股代码，或者直接要求分析某只股票的趋势。
2. Agent调用 Cassa 的 `business.py thises` 获取行情JSON。
3. Agent读取最近120条前复权日K。
4. Agent使用丁圣元《趋势投资》的方法判断趋势状态、阶段、形态、突破和风险。
5. Agent为主要方法性结论提供来源章节和经过核对的原文片段。

## 3. 设计目标

新建后的 Skill 必须形成以下闭环：

```text
用户趋势分析请求
  -> Skill被触发
  -> 解析股票代码
  -> 调用 business.py thises
  -> 读取完整 thises_input JSON
  -> 逐只分析日K
  -> 按任务加载知识模块
  -> 使用丁圣元框架形成结论
  -> 回到分章原文核对依据
  -> 按固定格式输出
  -> 执行自检
```

具体目标：

- 把 `SKILL.md` 作为用户请求进入趋势分析业务的唯一入口和流程控制器。
- 把 `business.py thises` 作为Skill调用的唯一日K数据获取方式，不把它定义为业务入口。
- 把书中总结、专题索引和分章原文作为按需加载的知识模块。
- 保证行情事实、分析判断、书本依据三者可以区分和追溯。
- 不让脚本替代趋势判断，也不让Agent虚构行情或原文。
- 不自动引入书外指标体系。

## 4. 非目标

第一版不处理以下事项：

- 不修改 `book-to-skill` 的通用生成流程。
- 不修改 `business.py`、`data.py` 或行情数据库。
- 不在 Skill 目录中增加新的Python脚本。
- 不使用或生成 `agents/openai.yaml`。
- 不生成 `source-manifest.json`。
- 不进行基本面、估值、新闻或政策分析。
- 不自动引入 MACD、RSI、KDJ、布林带、波浪理论、缠论等书外体系。
- 不使用 `chip` 字段形成趋势结论。
- 不负责自动下单或执行交易。
- 不把120条日K外推成多年或长期趋势结论。

## 5. 总体架构

### 5.1 职责划分

#### Cassa数据脚本

负责：

- 解析股票或板块代码。
- 从本地数据和实时行情中取得日K。
- 统一前复权口径。
- 清洗并按交易日期升序输出。
- 盘中合并或覆盖最新临时日K。
- 返回统一JSON。

#### `ding-trend-analyze` Skill

负责：

- 判断用户请求是否属于本Skill。
- 调用唯一数据命令。
- 读取并检查返回JSON。
- 从日K中识别高低点、趋势、横盘、缺口、形态和突破。
- 使用丁圣元框架解释行情。
- 按任务加载相应知识模块。
- 查找、核对并引用分章原文。
- 按固定报告结构输出并自检。

#### 知识文件

负责：

- `knowledge/` 提供快速规则、术语和专题路由。
- `summaries/` 提供章节理解与定位线索。
- `sources/` 提供最终可核对的分章原文。

### 5.2 判断边界

脚本只提供行情事实，不提供最终趋势结论。以下工作由Skill完成：

- 比较高点和低点。
- 判断上升、下降或横向状态。
- 判断趋势连贯性和发展阶段。
- 识别价格水平、趋势线、回撤和缺口。
- 识别持续形态和反转形态。
- 判断突破是已确认、待确认还是失败。

因此不增加 `computed_facts`。最高价、最低价和高低点关系属于Agent基于日K直接观察和比较的分析过程，不需要脚本预先给出结论。

### 10.3 标准输入结构

```json
{
  "task": "thises",
  "data_type": "thises_input",
  "created_at": "",
  "market_context": {
    "as_of": "",
    "is_intraday": null
  },
  "items": [
    {
      "raw_code": "",
      "target_type": "stock",
      "code": "",
      "name": "",
      "daily_kline": [
        {
          "code": "",
          "trade_date": "",
          "open_price": null,
          "high_price": null,
          "low_price": null,
          "close_price": null,
          "volume": null,
          "amount": null,
          "volume_ratio": null
        }
      ],
      "chip": {}
    }
  ],
  "errors": []
}
```

### 10.4 输入解释规则

- `task` 必须为 `thises`。
- `data_type` 必须为 `thises_input`。
- `items` 是逐只分析的唯一对象来源。
- 日K默认最近120条、前复权、按 `trade_date` 升序。
- 以实际收到的日K数量为准，不假设每只股票必然完整返回120条。
- `market_context.as_of` 是行情截止时间。
- `market_context.is_intraday=true` 时，最后一根K线尚未收盘，只能作为盘中观察，不能作为收盘确认。
- `market_context.is_intraday=false` 时，最后一根K线按完整交易日处理。
- `chip` 完全忽略。
- `errors` 不进入分析逻辑；如果 `items` 中没有对应可用对象，只说明没有取得可分析数据。
- `volume_ratio=null` 时忽略，不补算、不猜测。
- 不调用其他行情源补充或覆盖该JSON。

### 10.5 命令输出读取

当前Cassa的数据加载过程可能先输出进度日志，再输出最终JSON。因此不能假设整个标准输出从第一个字符开始就是纯JSON。

执行Agent应定位最终顶层对象，并同时确认其中存在：

```json
{
  "task": "thises",
  "data_type": "thises_input"
}
```

不得只读取JSON的开头、结尾或部分 `daily_kline`。如果工具输出被截断，应换用能够完整保留输出的读取方式后再分析，不能基于截断数据继续判断。

## 12. 标准业务流程

1. 从用户请求中提取一个或多个股票代码。
2. 如果没有可确定的代码，先要求用户提供代码，不猜测证券代码。
3. 在 `D:\股神养成plan\Cassa` 执行 `python business.py thises --codes <codes>`。
4. 读取完整命令输出并定位 `thises_input` JSON。
5. 检查 `task`、`data_type`、`market_context` 和 `items`。
6. 对 `items` 逐只处理；不得把不同股票的K线混合分析。
7. 读取 `chapter-router.md` 和 `cheatsheet.md`。
8. 根据行情实际出现的问题，按需读取 `patterns.md`、`glossary.md` 和相关章节总结。
9. 使用三层面顺序分析：观察范围内的整体趋势、价格形态、最新K线细节。
10. 判断市场状态、趋势连贯性、趋势阶段、关键位置、形态性质和突破状态。
11. 为主要方法性结论定位一个或多个相关章节。
12. 读取对应 `sources/*.txt`，使用 `rg -n -F` 取得章节内行号，核对上下文并摘录短原文。
13. 按固定格式逐只输出分析报告。
14. 执行完整自检后再交付。

## 13. 分析方法

### 13.1 数据范围

120条日K主要支持近期和中期趋势分析，包括：

- 观察范围内的高低点结构。
- 近期上涨、下跌或横向状态。
- 趋势推进与回撤的连贯性。
- 近期价格水平、支撑阻力和角色转换。
- 近期缺口、箱体、三角形、通道等价格结构。
- 最新突破、假突破和收盘确认。

120条日K不支持：

- 多年级别的长期趋势判断。
- 数据范围之外的重要历史高低点。
- 起点早于当前数据范围的完整大型形态。
- 需要周线、月线或分时数据才能确认的结论。

第七章要求全局、形态、细节三层分析，但当前只有120条日K。因此第一层只能表述为“本次数据范围内的整体结构”，不得冒充多年级别全局视角。

### 13.2 判断顺序

每只股票按以下顺序分析：

1. 数据范围与最后一根K线状态。
2. 观察范围内整体方向。
3. 标志性高点和低点是否抬高、降低或交错。
4. 趋势连贯性是否保持。
5. 当前更接近趋势三阶段中的哪一阶段；证据不足时写“阶段不清”。
6. 是否存在横向整理；若存在，判断更偏持续、反转还是未决。
7. 是否存在明显价格水平、趋势线、回撤、缺口或价格形态。
8. 最新信号是否获得收市确认。
9. 给出条件化的技术应对倾向和失效观察条件。

## 14. 约束与证据规则

### 14.1 丁圣元方法约束

以下约束来自书中框架，实施后要通过分章原文再次核对：

- 先判断方向和市场状态，再使用形态和细节工具。来源：引言、第三章、第八章。
- 用高点和低点的演变识别上涨、下跌或横向。来源：第二章、第三章。
- 单根K线不得脱离更大结构单独决定趋势。来源：第二章、第七章。
- 先看整体，再看形态，最后看最新K线细节。来源：第七章。
- 形态未突破关键边界前，保持未决，不提前押方向。来源：第三章、第五章、第六章。
- 盘中刺破不能自动视为已确认突破；收市和后续表现更重要。来源：第二章、第三章、第八章。
- 不能仅因涨幅大或跌幅大就断言反转。来源：第一章、第六章。
- 看不清时，不轻易否定原趋势。来源：第八章。
- 技术分析处理的是状态和可能前景，不是确定预测。来源：第三章、第九章。

### 14.2 系统完整性约束

以下属于Skill工程规则，不得表述成作者原话：

- 不捏造价格、日期、成交量、形态、行号或原文。
- 不引入JSON中不存在的行情数据。
- 不自动引入书外技术指标和理论。
- 不把章节总结当作原文。
- 不把盘中未完成K线当作收盘确认。

## 分析范围

- 近期价格水平、支撑阻力和角色转换。
- 近期缺口、箱体、三角形、通道和反转候选结构。

## 判断顺序

3. 比较标志性高点和低点是抬高、降低还是交错。
4. 判断趋势连贯性保持、减弱还是已经破坏。
5. 判断趋势阶段；证据不足时写“阶段不清”。
6. 若存在横向结构，判断更偏持续、反转还是未决。
7. 识别明显价格水平、趋势线、回撤、缺口和形态。
8. 判断最新信号是已收市确认、盘中待确认还是已经失败。
9. 给出条件化的技术应对倾向和后续观察条件。

- `quick_validate.py` 通过。
- 目标目录中只有运行必需文件。
- YAML只包含 `name` 和 `description`。
- 用户提出趋势分析时能够正确触发。
- 数据只来自 `business.py thises`。
- 默认120条、前复权、升序、盘中状态等约定得到正确执行。
- `chip` 和 `errors` 不进入趋势分析。
- 书中能力和工程规则分开表达。
- 总结文件只负责路由，原文文件负责证据。
- 核心结论具备行情证据、章节、原文文件、章节内起止行号和短原文。
- 每条原文片段都能在标注行号范围内逐字找到。
- 不出现书外指标、虚构数据、虚构行号、虚构引用或长期过度外推。
- 数据命令失败时明确停止，不生成伪分析。
