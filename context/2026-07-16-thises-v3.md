# Thises V3：量价关系归因法

## 量价关系归因法

### 一、目标

在 `coulling-volume-price-analysis` 量价分析 skill 中新增 `volume_price_relation` 模块，用“价格特征 + 买方/卖方候选原因 + 成交量验证”的方式，对最近几根 K 线进行量价关系归因。

该模块必须最终给出“看多”或“看空”的方向结论。这里的看多 / 看空是量价关系方向判断，不等同于直接的买入或卖出建议。

### 二、模块定位

当前量价分析流程调整为：

```text
输入日 K 数据
-> 量价关系归因
-> direction / truth / stage / smart_money / reversal / key_price
```

`volume_price_relation` 是基础归因模块，负责解释价格和成交量之间的关系；其他模块可以使用该模块的分析结果，但不应重复实现同一套局部量价归因逻辑。

各模块职责如下：

- `volume_price_relation`：识别价格特征，列出买方 / 卖方候选原因，用成交量验证，并给出看多 / 看空结论。
- `direction`：判断后续方向更偏多还是偏空。
- `truth`：判断价格运动是否得到成交量确认，还是存在量价异常。
- `stage`：根据更长历史 K 线切分吸筹、测试、拉升、派筹、下跌阶段。
- `smart_money`：判断更像吸收供给还是释放筹码。
- `reversal`：判断趋势可能反转的位置、异常条件和确认条件。
- `key_price`：分析关键支撑位和阻力位。

### 三、理论基础

该方法与以下量价分析框架相近：

- 威科夫投入—产出定律：成交量是投入，价格变化是产出，量价匹配属于确认，量价不匹配属于异常。
- 供求关系：价格结果需要结合需求和供给的相对变化解释。
- VPA / VSA：结合价格区间、实体、影线、收盘位置和成交量观察供给与需求。

本方法进一步把量价分析整理为一个最小归因流程：

```text
走势特征
-> 买方 / 卖方两个候选原因
-> 成交量变化验证
-> 选出更被支持的解释
-> 输出看多 / 看空
```

### 四、核心归因原则

#### 1. 先识别价格特征

不能只看单根 K 线，原则上比较最近连续 2～3 根或更多 K 线，观察：

- 方向：上涨、下跌、横盘；
- 速度：加速、放缓、平稳；
- 波动：扩大、收窄、剧烈震荡；
- K 线实体：实体扩大、实体缩小、实体稳定；
- 影线：上影线变长、下影线变长、影线收窄；
- 收盘位置：接近最高、接近最低、位于中部；
- 连续性：连续上涨、连续下跌、涨跌交替、趋势中断。

价格特征不限制为固定四种。模型应从这些观察维度中灵活组合出清晰的自然语言特征，例如：

```text
上涨趋势中的涨幅放缓，且阳线实体持续缩小
下跌趋势中的跌幅收窄，同时下影线变长
价格横盘，K 线实体缩小，成交量持续下降
价格上涨加速，但实体没有同步扩大
```

如果输入数据不足以识别价格特征，应降低结论强度，但仍必须给出看多或看空方向。

#### 2. 再列出两个候选原因

常见归因关系如下：

| 价格特征 | 候选原因 A | 候选原因 B |
|---|---|---|
| 涨幅放缓 | 买方减弱 | 卖方增加 |
| 涨幅加速 | 买方增加 | 卖方减弱 |
| 跌幅放缓 | 买方增加 | 卖方减弱 |
| 跌幅加速 | 买方减弱 | 卖方增加 |

对于横盘、影线变化、实体变化等复合特征，按照同样逻辑提出最接近的买方 / 卖方候选原因，不得强行套用固定形态名称。

#### 3. 只使用 `volume` 做验证

本模块只使用：

```text
daily_kline[].volume
```

本模块禁止使用：

```text
daily_kline[].volume_ratio
```

成交量观察包括：

- 放量；
- 缩量；
- 平量；
- 连续放量；
- 连续缩量；
- 先放量后缩量；
- 先缩量后放量。

`volume` 和 `volume_ratio` 的职责在本模块中明确分离：量价关系归因只看 `volume`；其他模块如有需要，可以按照各自协议使用 `volume_ratio`。

#### 4. 不能把成交量直接等同于买方或卖方

每一笔成交天然同时包含买方和卖方，因此：

```text
成交量放大 != 单独确认买方增加
成交量萎缩 != 单独确认买方减弱
```

成交量只能说明市场交易意愿、参与度或筹码交换强度发生变化。买方或卖方哪一方更占优势，必须结合价格方向、实体、影线、收盘位置和连续 K 线共同判断。

例如：

```text
上涨 + 放量 + 大实体
-> 更支持需求增强

上涨 + 放量 + 小实体或长上影
-> 更支持供给增加

下跌 + 缩量 + 跌幅收窄
-> 更支持供给减弱
```

### 五、看多 / 看空结论规则

模块必须输出看多或看空，不允许仅输出“无法判断”。如果证据不足，应输出倾向方向，并把结论强度或置信度降为弱。

基础规则如下：

| 价格表现 | 成交量表现 | 方向倾向 |
|---|---|---|
| 上涨加速 | 放量 | 看多 |
| 上涨加速 | 缩量 | 看多，但警惕虚涨 |
| 涨幅放缓 | 缩量 | 看空 |
| 涨幅放缓 | 放量 | 看空，警惕派发 |
| 下跌加速 | 放量 | 看空 |
| 下跌加速 | 缩量 | 看空，但下跌动能可能不足 |
| 跌幅放缓 | 缩量 | 看多 |
| 跌幅放缓 | 放量 | 根据实体、影线和后续价格结果选择方向，结论强度降低 |
| 横盘 | 缩量 | 中性偏多，等待测试；最终输出看多倾向 |
| 横盘 | 放量 | 中性偏空，警惕筹码交换或派发；最终输出看空倾向 |

结论应说明：

- 主导价格特征；
- 更支持哪一个候选原因；
- 成交量提供了什么证据；
- 为什么最终偏看多或看空；
- 证据是否充分。

例如：

```text
涨幅放缓，候选原因是买方减弱或卖方增加；成交量缩小，更支持买方交易意愿减弱，上涨动能衰减，因此量价关系暂时偏空，结论强度为中。
```

### 六、输出 JSON 结构

新增模块必须和其他普通模块保持一致，不增加特殊字段：

```json
"volume_price_relation": {
  "result": "",
  "data": [
    {
      "kline": {
        "code": "",
        "trade_date": "",
        "open_price": 0,
        "high_price": 0,
        "low_price": 0,
        "close_price": 0,
        "volume": 0,
        "amount": 0,
        "volume_ratio": 0
      },
      "evidence": ""
    }
  ],
  "refs": ["", ""]
}
```

其中：

- `result`：必须包含“看多”或“看空”，并说明结论强度；
- `data`：支撑判断的 K 线数据数组；
- `evidence`：说明价格特征、成交量变化、候选原因和方向结论；
- `refs`：填写对应的量价理论索引。

虽然本模块分析时只使用 `volume`，但输出的 `kline` 仍必须完整复制输入日 K 原始对象，包含 `volume_ratio` 字段，以符合现有 K 线引用协议。

### 七、盘中规则

如果 `market_context.is_intraday == true`：

- 最新一根 K 线的 `volume` 是盘中累计成交量；
- 不能把它与历史完整日 K 的全天成交量直接等价比较；
- 最新 K 的量价关系只能作为盘中暂态判断；
- `result` 和 `evidence` 中必须说明盘中限制；
- 可以输出看多或看空，但结论强度应适当降低；
- 不得写成已经确认突破、已经确认反转或已经完成放量止涨 / 止跌。

推荐表达：

```text
盘中暂时表现为量价偏多；由于最新成交量仍为累计值，收盘后仍需确认。
```

### 八、与现有模块的衔接

`volume_price_relation` 先完成局部量价归因，其他模块按照以下方式使用：

- `direction` 可以参考其看多 / 看空方向，但仍需结合完整历史 K 线；
- `truth` 可以使用其价格特征和成交量证据判断确认或异常；
- `stage` 不直接由单次量价关系决定，仍需根据历史区间切分阶段；
- `smart_money` 不能仅凭该模块的单次结论断言真实主力持仓，只能结合更多 K 线和筹码数据表达可能性；
- `reversal` 可以关注量价关系从看多转看空或从看空转看多的连续变化；
- `key_price` 仍根据输入 K 线的高低点和价格反复交换区域计算。

这里的“原始 JSON”指量价分析 skill 最终产生的分析结果 JSON，不是最初的 `thises_input` 数据包。

`report.py` 内部负责：

命名规则：


其中：

- `report_type`：报告类型，例如 `thesis`。
- `code`：来自最终分析 JSON 的 `code`。

- `data.json`：本次分析的原始结果 JSON。
- `report.md`：由同一份结果 JSON 渲染得到的 Markdown 报告。

日期和时间只生成一次，然后同时用于两个文件名。

## 2026-07-20 更新方案：盘中量价分析加入数据时间维度

### 一、问题背景

当前 thesis 量价分析只通过：

```json
"market_context": {
  "as_of": "2026-07-20 14:38:00",
  "is_intraday": true
}
```

中的 `market_context.is_intraday` 区分盘中和非盘中。

这个布尔标识只能说明最新一根日 K 是否处于盘中暂态，不能表达这根盘中 K 已经形成到什么程度。因此，以下两个时间点目前会进入同一套分析规则：

```text
09:38：开盘仅 8 分钟，成交量和 K 线形态都高度不完整。
14:38：全天 240 个交易分钟中已经完成 218 分钟，成交量已接近全天量级。
```

如果对二者一律使用“盘中成交量不能与历史全天成交量比较、统一降低结论强度”的规则，会产生两个问题：

1. 早盘数据可能被过度解读。
2. 尾盘数据的有效信息没有得到充分利用。

本次方案的目标是在不增加新数据字段、不修改 Python 数据采集代码的前提下，让 LLM 使用现有 `market_context.as_of` 判断盘中所处时间阶段，并据此调整成交量证据的权重。

### 二、设计原则

本次采用最小修改方案：

```text
is_intraday 决定最新 K 是否仍属于盘中暂态。
as_of 决定盘中数据当前处于早盘、普通盘中还是尾盘。
```

必须使用 `market_context.as_of`，不能要求 LLM 另外获取执行分析时的系统当前时间。

原因是 `market_context.as_of` 表示行情数据的实际采集时间，而 LLM 的执行时间可能晚于数据采集时间。例如：

```text
行情在 14:38 采集。
LLM 在 15:02 开始分析。
```

如果 LLM 使用 15:02 作为判断依据，就会把 14:38 的盘中临时 K 错误地当成收盘后完整 K。使用 `market_context.as_of` 可以保证时间口径与输入行情一致。

本次不新增以下字段：

```text
market_phase
session_progress
trading_minutes_elapsed
volume_completeness
```

这些字段以后如果需要更精细的确定性计算可以再增加；当前版本由 LLM 直接解析 `market_context.as_of` 即可满足需求。

### 三、盘中时间分段

当 `market_context.is_intraday == true` 时，LLM 必须从 `market_context.as_of` 中读取时分，并按以下三段处理：

| 时间段 | 阶段 | 成交量使用规则 | 结论强度 |
|---|---|---|---|
| 09:30 至 10:30 | 早盘 | 不得与历史完整日成交量直接比较，不得确认放量、缩量或平量 | 明显降低 |
| 10:30 至 14:30 | 普通盘中 | 可以观察累计量推进情况，但不得把低于历史全天量直接判断为缩量 | 适当降低 |
| 14:30 至 15:00 | 尾盘 | 可近似作为接近全天成交量，与历史完整日成交量做量级比较 | 可高于前两段，但仍非收盘确认 |

边界统一采用前闭后开，最后一个区间包含 15:00：

```text
09:30 <= time < 10:30
10:30 <= time < 14:30
14:30 <= time <= 15:00
```

由于 `is_intraday` 当前在午间为 `false`，上述 `10:30 至 14:30` 规则实际只会覆盖上午剩余交易时间和下午 13:00 后的交易时间，不会主动改变现有午间状态判断。本次不修改底层盘中状态函数。

### 四、尾盘成交量的非对称判断

盘中成交量是累计值，只会继续增加，不会减少。因此尾盘比较需要区分“已经达到参考量”和“仍低于参考量”：

```text
当前累计量 >= 历史参考量
```

可以明确表述为：

```text
当前累计成交量已经达到或超过历史参考量，量能放大已有较强盘中证据。
```

因为后续成交只会让最终成交量更大。

如果：

```text
当前累计量 < 历史参考量
```

则不能在收盘前严格确认缩量，只能表述为：

```text
当前尾盘累计量仍低于历史参考量，大概率表现为缩量，但最终结果仍需收盘确认。
```

本次不计算线性全天预测量，也不引入新的估算字段，避免把简单时间外推误当成真实全天成交量。

### 五、所有盘中阶段的共同限制

时间分段只调整盘中证据的可用程度，不改变“盘中 K 尚未定型”的基本事实。

无论处于早盘、普通盘中还是尾盘，都必须遵守：

1. 最新 `close_price` 是盘中当前价，不是最终收盘价。
2. 最新 `high_price`、`low_price`、实体和上下影线仍可能变化。
3. 不得输出“已经确认突破”“已经确认跌破”“已经完成反转”。
4. 不得输出“已经确认放量止涨”“已经确认缩量回踩”等收盘级结论。
5. 可以使用“早盘暂时出现”“普通盘中迹象”“尾盘迹象较强”“若收盘仍保持则可确认”等表达。
6. `volume_price_relation` 仍然只能使用 `daily_kline[].volume`，不得使用 `daily_kline[].volume_ratio` 作为归因依据。

````markdown
### 2. 普通盘中：10:30 至 14:30

```text

- 当前累计成交量可以近似视为接近全天成交量；
- 可以将其与历史完整日成交量进行量级比较；
- 如果当前累计成交量已经达到或超过历史参考量，可以明确说明量能已经达到或超过该参考量；

- 结论强度可以高于早盘和普通盘中，但仍不能等同于收盘确认。

推荐表达：

```text
````

## 2026-07-21 量价分析报告精简与“异常—测试—确认”模块新增方案

### 一、方案状态与目标

本章节是本轮量价分析报告整理的完整可执行方案。

当前状态：仅记录方案，尚未修改实际 skill、归因法和报告模板。

本次目标：

1. 删除重复、解释价值较低的报告内容。
2. 保留现有“反转判断”，新增独立的“异常—测试—确认”章节。
3. 要求报告正文中出现的每个价格都说明数据来源或推导来源。
4. 将量价关系统一改成“价格变化 → 成交量变化 → 买卖方结果 → 量价结论”的固定推理顺序。
5. 在量价关系的用户可见文字中使用“买方、卖方”，不再使用“供给、需求”等不直观表达。
6. 不修改 thises 原始数据采集、筹码计算和报告文件保存机制。

### 二、修改原因

#### 2.1 总体结论重复

当前模板中的“总体结论”和“一、量价关系/结论”都直接输出：

```jinja2
{{ data.volume_price_relation.result }}
```

两处内容完全相同，没有新增信息，因此删除“总体结论”。

#### 2.2 真实性判断与新模块职责重叠

当前 `truth` 模块用于判断价格运动是确认还是异常。新增“异常—测试—确认”后，异常识别、后续测试以及最终确认会形成完整证据链，已经覆盖 `truth` 的有效职责。

因此不能只从 Markdown 模板隐藏 `truth`，还应从 skill 输出 JSON 协议中删除，避免模型继续生成无用内容。

#### 2.3 阶段划分过长且容易强行归类

当前 `stage` 要求将整段历史 K 线划分为吸筹、测试、拉升、派筹、下跌。实际报告中容易出现：

1. 为了满足结构而强行切分阶段；
2. 阶段描述过长，挤占当前量价判断；
3. 与聪明钱、反转判断和关键价格重复引用相同历史 K 线。

因此删除 `stage` 模块和报告章节。

#### 2.4 价格缺少可追溯来源

当前报告会直接出现类似：

```text
反转观察区在 20.74—21.46 元。
收复 25.51—25.79 元才可能改善。
```

读者无法立即知道这些价格来自哪一天、哪个 K 线字段、筹码指标还是派生计算。新增价格来源规则后，正文中的价格必须可以回到输入 JSON 或明确的派生过程。

#### 2.5 量价关系表达不够直观

当前量价关系虽然已经包含买方/卖方归因，但输出顺序并不稳定，且仍会出现“供给增强”“需求减弱”等术语。

本次把推理顺序固定为：

```text
价格变化
→ 成交量变化
→ 买方力量/卖方压力的相对变化
→ 看多或看空结论
```

成交量只能说明交易活跃度和筹码交换强度，不能直接证明买家人数或卖家人数发生变化。因此最终用语必须是“买方力量增强/减弱”“卖方压力增强/减弱”或“更支持……”，不得断言“买家人数增加”“卖家人数减少”。

#### 2.6 新模块与反转判断需要明确边界

新增“异常—测试—确认”不替换现有“反转判断”。两者边界固定如下：

| 模块 | 时间方向 | 负责回答的问题 |
|---|---|---|
| 异常—测试—确认 | 回顾已经发生的 K 线 | 最近异常在哪里，后来有没有测试，测试后有没有确认或否定 |
| 反转判断 | 面向当前之后 | 当前是否接近反转，未来满足哪些价格和量能条件才算确认 |

`anomaly_test_confirmation` 不得重复输出未来反转价位计划；`reversal` 不得重新完整叙述已经发生的异常—测试—确认证据链。

##### 5.2.3 价格区间


##### 5.2.4 筹码价格

```text
8.50—12.10 元（筹码估算 90% 成本区间下沿—上沿）
```

筹码价格来自 `chip`，不虚构交易日期。

##### 5.2.5 派生或未来观察价

格式：

```text

```

“创新高”“突破位”“反转观察位”属于含义或用途，不是数据来源。必须同时说明它由哪个原始价格或计算规则推导。

### 六、“异常—测试—确认”模块规则

#### 6.1 模块职责

模块名称：

```text
anomaly_test_confirmation
```

模块只分析已经存在于输入 `daily_kline` 中的事实链：

```text
最近异常 K 线或连续异常 K 线
→ 异常之后是否出现测试
→ 测试之后是否出现确认或否定
```

#### 6.2 最近异常的选择

1. 按 `trade_date` 从近到远扫描输入 K 线。
2. 选择最近一根具备明确量价异常的 K 线。
3. 如果相邻多根 K 线共同构成同一种异常，允许作为一个连续异常组输出。
4. 不得把日期不连续、异常方向不同的 K 线拼成一个异常组。
5. 如果没有明确异常，输出 `anomaly.status = "not_found"`，不得为了填充报告强行指定异常。

#### 6.3 异常识别依据

异常的核心是价格结果与成交量投入不匹配。可识别的典型情况包括但不限于：

```text
成交量明显放大，但价格实体很小或收盘推进有限；
成交量明显放大，但出现长上影或长下影，说明盘中一方推进被另一方抵消；
价格大幅上涨或下跌，但成交量明显不足，运动缺少成交支持；
成交量继续放大，但连续 K 线的价格推进速度明显下降；
价格突破历史高点或低点，但成交量没有同步增强；
价格试图延续原方向，但收盘重新回到关键价格以内。
```

异常结论必须说明：

```text
价格发生了什么；
成交量发生了什么；
两者为什么不匹配；
更支持买方力量还是卖方压力发生变化；
异常偏多还是偏空。
```

#### 6.4 测试识别依据

测试只能出现在异常之后，并按时间顺序检查。测试可以由一根或连续多根 K 线组成。

典型测试包括：

```text
价格重新接近异常 K 线高点、低点或实体区间；
价格尝试延续异常前方向，但成交量降低且推进失败；
价格回踩异常区域，成交量降低且没有继续破位；
价格反抽异常区域，成交量降低且没有重新站稳；
价格短暂越过异常价格后又收回。
```

测试状态枚举：

```text
found
not_found
not_applicable
```

当没有异常时，测试必须为 `not_applicable`。

#### 6.5 确认识别依据

确认只能发生在异常和测试之后。确认用于判断异常所暗示的方向是否得到后续价格与成交量支持。

确认状态枚举：

```text
confirmed
invalidated
pending
not_applicable
```

定义：

| 状态 | 含义 |
|---|---|
| `confirmed` | 测试后价格沿异常暗示方向运行，且价格结果得到成交量支持 |
| `invalidated` | 后续价格明确反向运行，原异常解释被否定 |
| `pending` | 已有异常或测试，但后续 K 线不足以确认或否定 |
| `not_applicable` | 未发现异常，无法进入确认阶段 |

#### 6.6 与盘中数据的关系

继续读取 `market_context.is_intraday`。

当最后一根 K 线尚未收盘时：

1. 可以把它列为异常或测试的盘中候选证据。
2. 不得仅凭最后一根未收盘 K 线把 `confirmation.status` 写成 `confirmed` 或 `invalidated`。
3. 如果确认只依赖最后一根未收盘 K 线，状态写为 `pending`。
4. 模块只在涉及确认状态时说明“尚未收盘”，不在每个段落机械重复“盘中暂态”。

#### 6.7 成交量和量比使用范围

`volume_price_relation` 继续只能使用 `daily_kline[].volume`，继续禁止使用 `daily_kline[].volume_ratio`。

新增的 `anomaly_test_confirmation` 不属于 `volume_price_relation`，允许读取输入已经提供的 `volume_ratio` 作为补充证据，但必须遵守：

1. 历史完整 K 线优先使用连续 K 线的 `volume` 相对变化判断。
2. `volume_ratio` 只能补充说明相对活跃程度，不能单独证明买方或卖方变化。
3. 不重新计算 `volume_ratio`，直接使用 JSON 中的值。
4. 不讨论 `volume_ratio` 是实时接口值还是历史计算值，数据来源处理由上游程序负责。

### 二、修改原因

#### 2.1 三张表拆散了同一条时间证据链

当前“异常—测试—确认”分别输出：

```text
异常状态 + 异常结论 + 异常表格
测试状态 + 测试结论 + 测试表格
确认状态 + 确认结论 + 确认表格
```

这会把本来应按时间连续阅读的证据链拆成三段。读者需要来回对照日期，才能理解：

```text
哪一天出现异常
→ 哪一天进行了测试
→ 哪一天完成确认或否定
```

因此改成一个按 K 线日期从早到晚排列的表格。每行证据直接说明该 K 线为什么属于异常、测试或确认。

#### 2.2 盘中累计成交量不能完成横向比较

盘中最新 K 线的 `volume` 是截至当前时刻的累计成交量，而历史 K 线的 `volume` 是全天成交量。两者直接比较会把“尚未交易完一天”误判为缩量。

现有表达：

```text
2026-07-22 日累计成交量 27922860.0，属于盘中累计值，不能与完整日 K 的全天成交量直接比较。
```

这句话应保留，因为它准确说明了原始成交量的限制。但如果分析到这里就停止，盘中报告仍无法判断当前成交是否活跃。

#### 2.3 实时量比可以补足盘中成交活跃度判断

输入 JSON 已经为每根 K 线提供 `volume_ratio`，且上游程序负责历史值与实时值的生成。量价 skill 不需要重新计算，也不需要区分接口来源。

盘中可以继续输出原始累计成交量，同时比较：

```text
最新实时 K 线的 volume_ratio
与前 3～5 根 K 线各自的 volume_ratio
```

该比较回答的是“截至当前，今日相对成交活跃度高于还是低于近期”，不是“今日最终全天成交量是否放大”。

#### 2.4 当前归因法文件存在结构重复

当前 `量价关系归因法.md` 中存在：

```text
## 七、看多 / 看空规则
## 八、盘中规则
一段量价结论示例
## 七、看多 / 看空规则
## 八、盘中规则
```

标题重复且示例位于错误章节。实施本方案时必须一并整理为唯一的“七、看多/看空规则”“八、盘中规则”“九、输出要求”，避免新旧规则同时存在。

### 六、盘中量比比较规则

#### 6.1 字段职责

盘中分析中两个成交量字段的职责固定为：

| 字段 | 盘中用途 |
|---|---|
| `volume` | 原样报告截至当前的累计成交量，但不得与历史全天成交量直接比较 |
| `volume_ratio` | 比较最新实时量比与前 3～5 根 K 线量比，判断截至当前的相对成交活跃度 |

不得用 `volume_ratio` 替换或覆盖原始 `volume`。两者必须先后出现，各自回答不同问题。

#### 6.2 比较样本

当 `market_context.is_intraday == true` 时：

1. 最新一根 `daily_kline` 作为当前实时 K 线。
2. 从它之前按日期倒序选择最多 5 根、至少 3 根具有有效数值 `volume_ratio` 的 K 线。
3. 正常情况下比较前 5 根；为了避免文本过长，结论正文可以列出最近 3 根，并说明与前 5 根整体位置一致。
4. 如果历史有效样本只有 2 根，仍可比较，但必须说明样本较少并降低结论强度。
5. 如果历史有效样本少于 2 根，说明“历史量比样本不足，不能完成近期比较”，不得猜测。
6. 不重新计算任何一根 K 线的 `volume_ratio`。

#### 6.3 强弱判断

只做相对排序，不引入新的固定阈值：

| 最新实时量比的位置 | 成交活跃度判断 |
|---|---|
| 高于前 3～5 根全部值 | 明显高于近期 |
| 高于前 3～5 根多数值 | 高于近期多数交易日 |
| 位于近期值主要区间内 | 与近期大致相当 |
| 低于前 3～5 根多数值 | 低于近期多数交易日 |
| 低于前 3～5 根全部值 | 明显低于近期 |

不得仅因为 `volume_ratio > 1` 就直接写“放量确认”，也不得仅因为 `volume_ratio < 1` 就写“缩量确认”。必须同时比较前几根 K 线的实际量比值。

#### 6.4 输出顺序

盘中 `volume_price_relation.result` 的“成交量变化”必须依次包含：

```text
最新累计 volume 数值
→ 明确说明不能与历史全天 volume 直接比较
→ 最新实时 volume_ratio
→ 前 3～5 根 K 线的 volume_ratio 和日期
→ 当前相对成交活跃度结论
```

正确示例：

```text
成交量变化：2026-07-22 日累计成交量为 27922860.0，属于盘中累计值，不能与历史完整日 K 的全天成交量直接比较；今日实时量比为 1.46，高于 2026-07-21 日的 1.08、2026-07-20 日的 0.95 和 2026-07-17 日的 0.82，说明截至当前成交活跃度明显高于近期。
```

错误示例：

```text
今日成交量 27922860.0，较昨日缩量。
```

错误原因：把盘中累计量直接与历史全天量比较。

错误示例：

```text
今日量比 1.46，因此买方明显增加并确认放量上涨。
```

错误原因：量比只能说明相对活跃度，不能单独证明买方变化，也不能形成收盘级确认。

#### 6.5 买卖方归因

完成量比比较后，必须结合：

```text
价格方向；
实体大小；
上下影线；
最新价在盘中高低区间的位置；
连续 K 线的价格推进速度。
```

典型组合：

```text
价格上涨 + 接近盘中高位 + 实时量比高于近期
→ 更支持买方力量增强

价格上涨 + 长上影或收盘回落 + 实时量比高于近期
→ 更支持卖方压力同步增强，上涨推进被抵消

价格下跌 + 接近盘中低位 + 实时量比高于近期
→ 更支持卖方压力增强

价格窄幅波动 + 实时量比明显高于近期
→ 买卖双方交换增强，但暂不能唯一归因

价格上涨或下跌 + 实时量比低于近期
→ 当前价格运动的成交支持偏弱，降低方向结论强度
```

#### 6.6 盘后规则

当 `market_context.is_intraday == false` 时：

1. 最新 K 线的 `volume` 是完整日成交量，可以与历史完整 K 线直接比较。
2. `volume_price_relation` 继续使用 `volume` 做成交量归因。
3. 不要求使用 `volume_ratio`，避免同一信息重复表达。

因此这不是全面解除量比禁令，而是增加一个严格限定的盘中例外。
