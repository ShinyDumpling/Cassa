# Thises V3：量价关系归因法

## 量价关系归因法

### 一、目标

在 `coulling-volume-price-analysis` 量价分析 skill 中新增 `volume_price_relation` 模块，用“价格特征 + 买方/卖方候选原因 + 成交量验证”的方式，对最近几根 K 线进行量价关系归因。

该模块必须最终给出“看多”或“看空”的方向结论。这里的看多 / 看空是量价关系方向判断，不等同于直接的买入或卖出建议。

### 二、模块定位

当前量价分析流程调整为：

```text
输入日 K 数据
-> 量价关系归因
-> direction / truth / stage / smart_money / reversal / key_price
```

`volume_price_relation` 是基础归因模块，负责解释价格和成交量之间的关系；其他模块可以使用该模块的分析结果，但不应重复实现同一套局部量价归因逻辑。

各模块职责如下：

- `volume_price_relation`：识别价格特征，列出买方 / 卖方候选原因，用成交量验证，并给出看多 / 看空结论。
- `direction`：判断后续方向更偏多还是偏空。
- `truth`：判断价格运动是否得到成交量确认，还是存在量价异常。
- `stage`：根据更长历史 K 线切分吸筹、测试、拉升、派筹、下跌阶段。
- `smart_money`：判断更像吸收供给还是释放筹码。
- `reversal`：判断趋势可能反转的位置、异常条件和确认条件。
- `key_price`：分析关键支撑位和阻力位。

### 三、理论基础

该方法与以下量价分析框架相近：

- 威科夫投入—产出定律：成交量是投入，价格变化是产出，量价匹配属于确认，量价不匹配属于异常。
- 供求关系：价格结果需要结合需求和供给的相对变化解释。
- VPA / VSA：结合价格区间、实体、影线、收盘位置和成交量观察供给与需求。

本方法进一步把量价分析整理为一个最小归因流程：

```text
走势特征
-> 买方 / 卖方两个候选原因
-> 成交量变化验证
-> 选出更被支持的解释
-> 输出看多 / 看空
```

### 四、核心归因原则

#### 1. 先识别价格特征

不能只看单根 K 线，原则上比较最近连续 2～3 根或更多 K 线，观察：

- 方向：上涨、下跌、横盘；
- 速度：加速、放缓、平稳；
- 波动：扩大、收窄、剧烈震荡；
- K 线实体：实体扩大、实体缩小、实体稳定；
- 影线：上影线变长、下影线变长、影线收窄；
- 收盘位置：接近最高、接近最低、位于中部；
- 连续性：连续上涨、连续下跌、涨跌交替、趋势中断。

价格特征不限制为固定四种。模型应从这些观察维度中灵活组合出清晰的自然语言特征，例如：

```text
上涨趋势中的涨幅放缓，且阳线实体持续缩小
下跌趋势中的跌幅收窄，同时下影线变长
价格横盘，K 线实体缩小，成交量持续下降
价格上涨加速，但实体没有同步扩大
```

如果输入数据不足以识别价格特征，应降低结论强度，但仍必须给出看多或看空方向。

#### 2. 再列出两个候选原因

常见归因关系如下：

| 价格特征 | 候选原因 A | 候选原因 B |
|---|---|---|
| 涨幅放缓 | 买方减弱 | 卖方增加 |
| 涨幅加速 | 买方增加 | 卖方减弱 |
| 跌幅放缓 | 买方增加 | 卖方减弱 |
| 跌幅加速 | 买方减弱 | 卖方增加 |

对于横盘、影线变化、实体变化等复合特征，按照同样逻辑提出最接近的买方 / 卖方候选原因，不得强行套用固定形态名称。

#### 3. 只使用 `volume` 做验证

本模块只使用：

```text
daily_kline[].volume
```

本模块禁止使用：

```text
daily_kline[].volume_ratio
```

成交量观察包括：

- 放量；
- 缩量；
- 平量；
- 连续放量；
- 连续缩量；
- 先放量后缩量；
- 先缩量后放量。

`volume` 和 `volume_ratio` 的职责在本模块中明确分离：量价关系归因只看 `volume`；其他模块如有需要，可以按照各自协议使用 `volume_ratio`。

#### 4. 不能把成交量直接等同于买方或卖方

每一笔成交天然同时包含买方和卖方，因此：

```text
成交量放大 != 单独确认买方增加
成交量萎缩 != 单独确认买方减弱
```

成交量只能说明市场交易意愿、参与度或筹码交换强度发生变化。买方或卖方哪一方更占优势，必须结合价格方向、实体、影线、收盘位置和连续 K 线共同判断。

例如：

```text
上涨 + 放量 + 大实体
-> 更支持需求增强

上涨 + 放量 + 小实体或长上影
-> 更支持供给增加

下跌 + 缩量 + 跌幅收窄
-> 更支持供给减弱
```

### 五、看多 / 看空结论规则

模块必须输出看多或看空，不允许仅输出“无法判断”。如果证据不足，应输出倾向方向，并把结论强度或置信度降为弱。

基础规则如下：

| 价格表现 | 成交量表现 | 方向倾向 |
|---|---|---|
| 上涨加速 | 放量 | 看多 |
| 上涨加速 | 缩量 | 看多，但警惕虚涨 |
| 涨幅放缓 | 缩量 | 看空 |
| 涨幅放缓 | 放量 | 看空，警惕派发 |
| 下跌加速 | 放量 | 看空 |
| 下跌加速 | 缩量 | 看空，但下跌动能可能不足 |
| 跌幅放缓 | 缩量 | 看多 |
| 跌幅放缓 | 放量 | 根据实体、影线和后续价格结果选择方向，结论强度降低 |
| 横盘 | 缩量 | 中性偏多，等待测试；最终输出看多倾向 |
| 横盘 | 放量 | 中性偏空，警惕筹码交换或派发；最终输出看空倾向 |

结论应说明：

- 主导价格特征；
- 更支持哪一个候选原因；
- 成交量提供了什么证据；
- 为什么最终偏看多或看空；
- 证据是否充分。

例如：

```text
涨幅放缓，候选原因是买方减弱或卖方增加；成交量缩小，更支持买方交易意愿减弱，上涨动能衰减，因此量价关系暂时偏空，结论强度为中。
```

### 六、输出 JSON 结构

新增模块必须和其他普通模块保持一致，不增加特殊字段：

```json
"volume_price_relation": {
  "result": "",
  "data": [
    {
      "kline": {
        "code": "",
        "trade_date": "",
        "open_price": 0,
        "high_price": 0,
        "low_price": 0,
        "close_price": 0,
        "volume": 0,
        "amount": 0,
        "volume_ratio": 0
      },
      "evidence": ""
    }
  ],
  "refs": ["", ""]
}
```

其中：

- `result`：必须包含“看多”或“看空”，并说明结论强度；
- `data`：支撑判断的 K 线数据数组；
- `evidence`：说明价格特征、成交量变化、候选原因和方向结论；
- `refs`：填写对应的量价理论索引。

虽然本模块分析时只使用 `volume`，但输出的 `kline` 仍必须完整复制输入日 K 原始对象，包含 `volume_ratio` 字段，以符合现有 K 线引用协议。

### 七、盘中规则

如果 `market_context.is_intraday == true`：

- 最新一根 K 线的 `volume` 是盘中累计成交量；
- 不能把它与历史完整日 K 的全天成交量直接等价比较；
- 最新 K 的量价关系只能作为盘中暂态判断；
- `result` 和 `evidence` 中必须说明盘中限制；
- 可以输出看多或看空，但结论强度应适当降低；
- 不得写成已经确认突破、已经确认反转或已经完成放量止涨 / 止跌。

推荐表达：

```text
盘中暂时表现为量价偏多；由于最新成交量仍为累计值，收盘后仍需确认。
```

### 八、与现有模块的衔接

`volume_price_relation` 先完成局部量价归因，其他模块按照以下方式使用：

- `direction` 可以参考其看多 / 看空方向，但仍需结合完整历史 K 线；
- `truth` 可以使用其价格特征和成交量证据判断确认或异常；
- `stage` 不直接由单次量价关系决定，仍需根据历史区间切分阶段；
- `smart_money` 不能仅凭该模块的单次结论断言真实主力持仓，只能结合更多 K 线和筹码数据表达可能性；
- `reversal` 可以关注量价关系从看多转看空或从看空转看多的连续变化；
- `key_price` 仍根据输入 K 线的高低点和价格反复交换区域计算。

## 2026-07-16 更新：thesis 调用 report 输出 JSON 和 Markdown

### 一、目标

量价分析 skill 完成结构化 JSON 分析结果后，调用通用 `report.py`，同时保存：

1. 本次分析的原始 JSON 结果。
2. 根据同一份 JSON 渲染得到的 Markdown 报告。

两个文件都必须写入：

```text
D:\股神养成plan\Cassa\result
```

### 二、整体流程

```text
thises-volume-price skill
    -> 调用 business.py thises 获取 thises_input JSON
    -> 调用 coulling-volume-price-analysis skill
    -> 得到最终 thesis 分析 JSON
    -> 调用 report.py
    -> report.py 生成同一批次的 JSON 文件和 Markdown 文件
    -> 两个文件写入 result/
```

这里的“原始 JSON”指量价分析 skill 最终产生的分析结果 JSON，不是最初的 `thises_input` 数据包。

### 三、调用边界

thesis 不负责自己拼接输出文件名，也不负责分别保存 JSON 和 Markdown。

thesis 只需要把最终分析 JSON 对象和报告类型传给 `report.py`：

```python
from report import generate_report_bundle

result_paths = generate_report_bundle(
    data=thesis_analysis_result,
    report_type="thesis",
)
```

`report.py` 内部负责：

1. 生成本次运行唯一的批次标识。
2. 根据 JSON 中的 `code` 和 `name` 生成文件名主体。
3. 先把 JSON 原样序列化到 `result/`。
4. 使用同一份数据和同一份批次标识渲染 Markdown。
5. 把 Markdown 写入 `result/`。
6. 返回两个文件的路径。

这样可以保证 JSON 和 Markdown 一定属于同一次报告生成，不会出现两个文件日期不同或标的名称不同的问题。

### 四、文件命名方案

两个文件使用完全相同的主体名称，只通过后缀区分用途：

```text
thesis-002546.SZ-新联电子-20260716-145045-data.json
thesis-002546.SZ-新联电子-20260716-145045-report.md
```

命名规则：

```text
{report_type}-{code}-{name}-{run_timestamp}-data.json
{report_type}-{code}-{name}-{run_timestamp}-report.md
```

其中：

- `report_type`：报告类型，例如 `thesis`。
- `code`：来自最终分析 JSON 的 `code`。
- `name`：来自最终分析 JSON 的 `name`。
- `run_timestamp`：本次生成报告的时间，格式为 `YYYYMMDD-HHMMSS`。
- `data.json`：本次分析的原始结果 JSON。
- `report.md`：由同一份结果 JSON 渲染得到的 Markdown 报告。

日期和时间只生成一次，然后同时用于两个文件名。

### 五、禁止覆盖规则

每次生成报告都必须产生新的文件，不允许覆盖已有文件。

生成批次标识时，优先使用：

```text
YYYYMMDD-HHMMSS
```

如果同一秒内目标文件已经存在，则自动增加微秒或递增序号，例如：

```text
thesis-002546.SZ-新联电子-20260716-145045-123456-data.json
thesis-002546.SZ-新联电子-20260716-145045-123456-report.md
```

注意：判断是否冲突时，必须以两个文件的共同主体为单位。不能只检查 JSON 文件而忽略 Markdown 文件，也不能让两个文件使用不同的备用时间戳。

文件写入建议使用独占创建模式：

```python
open(path, "x", encoding="utf-8")
```

如果文件已经存在，则重新生成一组新的共同文件名。

### 六、写入顺序

为了避免 Markdown 成功但 JSON 保存失败，或者 JSON 保存后模板渲染失败，执行顺序如下：

```text
1. 校验 data 是 dict。
2. 校验 report_type 对应的模板存在。
3. 在内存中完成 Markdown 渲染。
4. 生成一组不冲突的共同文件名。
5. 将 data 原样写入 data.json。
6. 将已经渲染好的 Markdown 写入 report.md。
7. 返回两个文件路径。
```

如果第 6 步写入失败，应删除第 5 步已经写入的 JSON 文件，避免 `result/` 中留下只有数据、没有报告的不完整批次。

### 七、JSON 保存规则

保存 JSON 时不改变分析结果内容，只进行格式化序列化：

```python
json.dump(
    data,
    file,
    ensure_ascii=False,
    indent=2,
)
```

要求：

- 保留中文。
- 使用 2 个空格缩进。
- 文件末尾保留换行。
- 不在保存阶段新增字段。
- 不在保存阶段修改分析结论。

### 八、Markdown 生成规则

Markdown 使用和 `2026-07-16-report.md` 中一致的 `templates/thesis.md.j2` 模板。

报告结构到“筹码分析”结束：

```text
# 标的名称（代码）量价分析报告
## 分析上下文
## 总体结论
## 一、量价关系
## 二、方向判断
## 三、真实性判断
## 四、阶段划分
## 五、聪明钱
## 六、反转判断
## 七、关键价格
## 八、筹码分析
```

不再增加：

```text
## 九、报告说明
```

盘中提示只根据：

```text
market_context.is_intraday
```

输出固定的盘中暂态提示。模板不根据多个字段重新推导新的业务结论。

### 九、建议的 report.py 对外接口

第一版将原来的单文件输出接口调整为一个报告批次接口：

```python
def generate_report_bundle(
    data: dict,
    report_type: str,
) -> dict:
    """保存原始 JSON 和 Markdown 报告，返回两个文件路径。"""
```

不保留 `output_name` 参数。文件名统一由 `report.py` 根据报告类型、代码、名称和本次运行时间自动生成，避免调用方绕过统一的批次命名和不覆盖规则。

返回结构：

```python
{
    "data_path": Path(".../result/...-data.json"),
    "report_path": Path(".../result/...-report.md"),
}
```

如果调用方需要向用户展示结果，可以使用返回的两个路径；调用方不需要再自行拼接文件名或重复写文件。
