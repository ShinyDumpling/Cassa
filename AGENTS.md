# Cassa 项目 Agent 规则

以下规则适用于本项目中的所有 Agent、LLM 与会话。

## 目录规则

1. 所有中间产物，包括但不限于 `json`、`md`、临时 `py` 脚本等，必须放在与 `cassa.py` 同级的 `tmp/` 目录中。
2. 所有最终结果产物必须放在与 `cassa.py` 同级的 `result/` 目录中。
3. 所有数据源必须放在与 `cassa.py` 同级的 `data/` 目录中。

## 协作规则

1. 在修改任何代码之前，必须先与用户讨论修改思路和修改方案，并获得确认。未经确认，不得直接修改代码。
2. 未经用户明确允许，不得提交代码。
3. 每当用户要求“检查分支”或表达相同含义时，必须先检查：
   - 当前项目有多少个分支
   - 当前所在分支是什么
   - 当前有哪些未提交文件
4. `TASKS.md` 是本项目唯一的任务清单入口。开始执行具体任务前，应先阅读 `TASKS.md`，确认当前任务状态和上下文。

## 代码编写规则

### 可读性

1. 变量名和函数名必须让人一眼看出用途，禁止使用含义不清的名字，例如：`a`、`b`、`tmp1`、`data2`、`do_stuff`。
2. 优先使用完整、明确、可读的英文命名。宁可稍长，也不要为了简短牺牲可读性。
3. 同一类概念在整个项目中必须统一命名，不能同一个东西一会儿叫 `context`，一会儿叫 `payload`，一会儿叫 `data`。
4. 一个函数只做一件事，函数名必须准确描述这件事。
5. 代码的阅读顺序应尽量贴近执行顺序，减少必须跳来跳去才能看懂的情况。

### 复用优先

1. 新增函数前，必须先扫描项目中是否已有可复用的函数。
2. 优先使用已有函数，或在此基础上修改旧函数以适配新需求，而不是另起炉灶新建同名或近义函数。
3. 如果旧函数需要小改动就能复用，优先改旧函数；只有当改动会导致旧函数职责膨胀或影响已有调用方时，才考虑新建。

### 注释

1. 每个函数都必须写注释或 docstring，至少说明：
   - 这个函数是做什么的
   - 输入是什么
   - 输出是什么
2. 所有注释和 docstring 默认必须使用中文；只有外部接口名、库名、协议名、字段名等必须保留原文时，才允许在中文注释中夹带英文术语。
3. 关键变量、路径变量、状态变量、重要中间变量必须写注释。
4. 普通短生命周期局部变量不强制逐个写注释，但变量名本身必须足够清晰。
5. 注释应优先解释“为什么这样做”或“这段逻辑在流程中的作用”，不要只重复代码表面意思。
6. 如果一段逻辑不容易一眼看懂，可以在代码块前增加一行简短说明。
7. 禁止写低价值废话注释，例如“给变量赋值”“调用函数获取结果”。

### 文件结构

1. 单个脚本内部应尽量按固定顺序组织：
   - 文件头说明
   - import
   - 常量 / 路径 / 配置
   - 数据结构定义
   - 工具函数
   - 核心流程函数
   - 主入口 `main()`
2. 路径、默认配置、开关项优先集中定义，避免散落在代码各处硬编码。
3. 先定义函数，再在文件末尾组织主流程入口，不要边写逻辑边临时定义函数。
4. 优先保持一个明确主入口，不要无必要制造多个隐式入口。
5. 即使当前约定所有代码先写在一个 `.py` 文件中，也必须按“可分区、可拆分”的方式组织，不能把不同职责的逻辑混写成一整坨。
6. 单文件内建议按功能分区组织，例如：
   - 配置与常量区
   - 数据结构区
   - 股票代码规范化区
   - 外部接口接入区
   - 数据整理区
   - 业务流程区
   - 输出区
   - 主入口区
7. 纯配置放常量区，纯数据优先用 `dataclass` 或等价清晰结构表达，无状态工具优先写成函数；只有在“需要维护状态”或“一组方法天然围绕同一个对象”时才使用类。
8. 禁止为了“分类好看”机械创建大而全的工具类、管理类、静态方法容器类，例如把大量无状态函数全部塞进 `XXUtils`、`XXManager`、`XXService` 这一类空泛命名中。
9. 第一版默认遵循“少量必要类 + 清晰函数 + 清晰数据结构”的原则，不为了形式上的面向对象而过度设计。

### 输出与调试

1. 控制台输出必须分层：
   - 先输出最重要的摘要
   - 再输出必要的过程信息
   - 调试细节仅在 debug 模式下输出
2. 默认不打印大段原始 JSON、大段 Prompt、大段模型原始响应。
3. 中间产物默认不落盘；只有 debug 模式才允许输出到 `tmp/`。
4. 最终结果产物必须输出到 `result/`，命名要清晰、稳定、可追溯。
5. 最终结果文件默认不覆盖旧结果，文件名中应加入日期时间或其他可区分标识。

### 修改方式

1. 讨论修改思路和修改方案、以及真正修改代码时，都要遵循最小改动原则。
2. 新增功能时，优先补到现有主流程中，不要一上来就拆出新的脚本或新的模块。
3. 每次修改都要尽量小而清晰，避免一次性混入很多不相关改动。
4. 如果某个改动会影响目录结构、输入输出格式或文件命名，必须先说明影响，再进行修改。

### 风格与复杂度控制

1. 优先写直白代码，不要为了“优雅”或“高级感”引入过度抽象。
2. 第一版优先可读、可跑、可改，不优先追求抽象和复用。
3. 没有必要时不要引入复杂类；能用清晰函数解决的，优先用函数。
4. 如果某个能力暂时不确定，先留 `TODO`，不要提前做复杂设计。
5. 不允许为了“以后可能复用”而提前拆分过多模块。

### 稳健性

1. 关键步骤必须有清晰错误提示，报错信息要让人一眼知道卡在哪一步。
2. 所有外部数据读取都必须考虑缺失、为空、格式不对、数据过期等情况，不能默认输入永远正确。

## Token 与上下文控制规则

1. 对于任何输出大小未知或可能很大的命令，必须先限制输出大小。优先按字节数控制，不要只按行数控制。
2. 不允许直接读取大日志、大 JSON、大 SQLite、大二进制文件、大 base64 文件的完整内容。
3. 默认只查看错误摘要、关键 warning、关键片段，不默认读取完整日志。
4. 修改后优先执行最小必要验证，不要机械执行全量测试、全量构建、全量扫描。
5. 对未知文件，先判断文件类型和文件大小，再决定是否读取内容。
6. 中间调试信息、原始模型响应、完整 Prompt、完整上下文默认不进入主流程输出，只有 debug 模式才允许保留和查看。
7. 上下文质量优先于上下文数量，避免把无关文本、大段日志、重复输出带入当前会话。

## 面向 Agent 的额外要求

1. 在新增代码前，应先阅读本文件和 `TASKS.md`。
2. 新增代码时，必须尽量保持现有命名风格、输出风格、目录约定一致。
3. 如果存在“更复杂的实现”和“更简单的实现”，第一版默认选更简单的实现。
4. 当项目在多个 Agent、多个会话之间切换时，后续 Agent 必须主动对齐已有代码的命名风格、注释风格、输出风格、函数组织方式，以及分析问题的逻辑口径和思考角度，避免不同 Agent 各写各的、各想各的，导致项目内部风格和业务理解割裂。
5. 在修改已有代码或新增代码之前，必须先阅读并理解项目中现有的相关代码（包括旧项目中被整合进来的原始代码），以其书写习惯、代码逻辑模式和解决问题的思路作为参考基准。不得在风格、命名、结构组织上引入与现有代码不一致的新模式，除非有明确的理由且已在讨论阶段获得确认。

## API Key 安全原则

1. 任何真实 API key 禁止写入仓库文件。
2. 日志里禁止打印完整 API key。
3. 调试输出里如果涉及 key，只允许显示前几位和后几位，中间必须脱敏。
4. 任何真实 API key 禁止写入数据库。
